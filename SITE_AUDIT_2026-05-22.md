# Аудит сайта smirnovads.com (2026-05-22)

## Что проверено
- Структура репозитория: статический сайт (HTML/CSS/JS), есть разделы Home, Cases, Blog, Privacy.
- Технические файлы: `robots.txt`, `sitemap.xml`, `_headers`.
- Базовые SEO/UX/контент/поддерживаемость в контексте редактирования через AI-ассистентов.

## Сильные стороны
- Хорошо оформленная главная с понятным оффером и сильным social proof (кейсы, цифры, география).
- Есть каноникал, OG, Twitter Card и JSON-LD на главной.
- Есть страница политики конфиденциальности и cookie-баннер.
- Есть блог и страница кейсов, а также `sitemap.xml` и `robots.txt`.
- Навигация и футер централизованы через `assets/js/nav.js` (удобно для поддержки).

## Риски и точки роста

### 1) Инлайн-стили и дублирование CSS
На главной много инлайн-стилей и большой embedded CSS в `<style>`, параллельно с `assets/css/main.css`.
- Риск: AI-правки становятся менее предсказуемыми (часть стилей в одном месте, часть в другом), растет вероятность конфликтов.
- Рекомендация: вынести page-specific стили в отдельный файл (например, `assets/css/home.css`) и подключать после `main.css`.

### 2) Неполное покрытие структурированными данными
JSON-LD есть на главной, но не видно унифицированного шаблона для кейсов/статей.
- Рекомендация:
  - Для блога: `BlogPosting` на каждой статье.
  - Для кейсов: `Article` или `CreativeWork` + поля `datePublished`, `author`, `headline`, `about`.
  - Для сайта в целом: `Organization` с контактами и профилями.

### 3) Нет явного editorial-процесса для AI-правок
Репозиторий минималистичный, но нет отдельного файла правил для AI-редакторов (типа `AGENTS.md`/`CONTRIBUTING.md`).
- Риск: разные ассистенты меняют структуру по-разному.
- Рекомендация: добавить `AGENTS.md` с четкими правилами:
  - что можно менять автоматически, а что только вручную;
  - стандарты мета-тегов;
  - чеклист перед коммитом;
  - naming convention для новых страниц.

### 4) Ограниченные security-заголовки
В `_headers` есть базовые заголовки, но нет явного CSP и некоторых дополнительных защит.
- Рекомендация: постепенно внедрить
  - `Content-Security-Policy` (с учетом Яндекс.Метрики),
  - `Permissions-Policy`,
  - `Strict-Transport-Security` (если всегда HTTPS).

### 5) Точечная проверка кэширования
В `_headers` указан `/assets/main-photo.jpeg`, а в проекте используется `assets/main-photo.webp`.
- Рекомендация: синхронизировать правило кэша с фактическим расширением файла.

### 6) Контентная воронка
Сильный персональный бренд уже есть, но можно увеличить конверсию:
- Добавить «мини-офферы» в блоге (CTA после 30–40% статьи).
- Добавить «что получите за 30 минут консультации» (bullet-формат).
- В кейсах сделать блоки «исходные данные → действия → результат → сроки» единым шаблоном.

## Что добавить в ближайшие 2 недели (приоритет)
1. **AGENTS.md + CONTRIBUTING чеклист** для AI-редактирования.
2. **Вынести стили главной** в отдельный CSS-файл.
3. **Исправить кэш-правило** для `main-photo.webp` в `_headers`.
4. **Добавить JSON-LD для blog posts** минимум на 2 текущие статьи.
5. **Добавить лид-магнит CTA** в статьи блога.

## Предложение по правилам для AI (черновик)
- Любая новая страница обязана иметь: `title`, `description`, `canonical`, OG-теги.
- Любая статья блога обязана иметь `BlogPosting` JSON-LD.
- Запрещено добавлять инлайн-стили, если можно использовать существующие utility-классы.
- Перед коммитом: проверить ссылки, sitemap, и отсутствие дублированных H1.

## Заключение
Сайт уже выглядит профессионально и «продающе». Следующий шаг — сделать его **операционно устойчивым для AI-редактирования**: формализовать правила, уменьшить дублирование стилей и усилить SEO-структуру на уровне шаблонов.
