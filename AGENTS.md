# AGENTS.md — правила редактирования сайта через Claude/Codex

Область действия: весь репозиторий.

## 1) Цель документа
Этот файл задаёт единые правила для AI-ассистентов (Claude/Codex), чтобы правки были:
- предсказуемыми,
- безопасными для SEO и продакшена,
- удобными для ревью человеком.

---

## 2) Общие принципы работы
1. Делай **минимально достаточные** изменения под задачу (без лишних рефакторингов).
2. Сохраняй текущую архитектуру статического сайта (HTML/CSS/JS без фреймворков), если явно не просят иначе.
3. Не меняй тексты/офферы/позиционирование без явного запроса.
4. Не удаляй рабочий функционал ради «красоты кода».
5. Если задача неочевидна — добавь краткие комментарии в PR (что изменено и почему).

---

## 3) SEO/мета-правила (обязательно)
Для каждой новой страницы обязательно:
- уникальный `<title>`;
- уникальный `<meta name="description">`;
- `<link rel="canonical" ...>`;
- Open Graph минимум: `og:type`, `og:url`, `og:title`, `og:description`, `og:image`;
- корректный `twitter:card`.

Для контентных страниц:
- блог: добавлять `BlogPosting` JSON-LD;
- кейсы: добавлять `Article` или `CreativeWork` JSON-LD.

После добавления/удаления страниц:
- обновить `sitemap.xml` (URL + `lastmod`),
- проверить `robots.txt` (актуальная ссылка на sitemap).

---

## 4) Пост-деплой для контентных страниц
После деплоя новой статьи или кейса подготовить в чате кросспостинг-пакет:
- версию для Telegram: короткий пост, с сильным первым абзацем, без простыни;
- версию для порталов вроде vc.ru и LinkedIn: более строгий самостоятельный текст, отличающийся от версии на сайте;
- тематическое изображение для поста/портала, если пользователь явно не просит без изображения.

Правила для ссылок в кросспостинге:
- в конце Telegram-поста добавлять фразу-ссылку: `Ссылку на полную статью оставлю здесь`;
- всю фразу `Ссылку на полную статью оставлю здесь` делать ссылкой на полную статью с UTM-метками:
  `utm_source=telegram`, `utm_medium=social`, `utm_campaign=article`,
  `utm_content=article_DD-MM_slug`;
- `DD-MM` — день и месяц публикации статьи без года, `slug` — английский slug страницы из URL без начальных/конечных слешей и без раздела `blog`, например:
  `article_03-06_deshevye-lidy-necelevye-b2b`;
- для vc.ru и LinkedIn в конце добавлять: `Веду канал TG: Заметки дорогого квала. Еще кейсы`;
- в этой фразе `Заметки дорогого квала` делать ссылкой на Telegram-канал `https://t.me/qual_notes`;
- `Еще кейсы` делать ссылкой на сайт `https://smirnovads.com/cases/`.

Требования к изображению:
- оно должно раскрывать тему материала, а не быть абстрактной иллюстрацией;
- на изображении должен присутствовать Алексей Смирнов;
- перед генерацией или подбором изображения уточнить, какой референс/фото использовать, если подходящего актуального изображения не задано явно;
- не менять `og:image` статьи на это изображение без отдельного запроса.

---

## 5) Правила по HTML/CSS/JS
1. Предпочитать существующие стили и паттерны из `assets/css/main.css` и текущих шаблонов.
2. Не добавлять инлайн-стили, если можно решить через класс.
3. Не дублировать крупные CSS-блоки между страницами.
4. Не подключать новые библиотеки JS без явного запроса.
5. Любой новый JS должен быть простым, без скрытых сайд-эффектов и без поломки существующей навигации/модалок.

---

## 6) Privacy/analytics/security
1. Не удалять и не ломать Яндекс.Метрику без явного запроса.
2. Если меняются формы/сбор данных — убедиться, что это не противоречит `privacy/index.html`.
3. При работе с `_headers`:
   - не ослаблять существующие заголовки безопасности;
   - кэш-правила должны соответствовать фактическим расширениям файлов.

---

## 7) Git и формат изменений
1. Один PR = одна логическая задача.
2. Commit message: коротко и по делу (`feat: ...`, `fix: ...`, `docs: ...`).
3. В PR-описании обязательно:
   - что сделано;
   - зачем;
   - какие файлы затронуты;
   - как проверяли.

---

## 8) Чеклист перед завершением задачи
Перед финалом AI-ассистент обязан проверить:
- [ ] Нет сломанных внутренних ссылок.
- [ ] На странице ровно один `h1`.
- [ ] `title`/`description`/`canonical` заполнены корректно.
- [ ] Для новых материалов обновлён `sitemap.xml`.
- [ ] После деплоя контентной страницы подготовлен кросспостинг-пакет или явно указано, что деплой ещё не выполнялся.
- [ ] Не добавлены случайные инлайн-стили и дубли CSS.
- [ ] Изменения минимальны и соответствуют задаче.

---

## 9) Что запрещено делать без явного запроса
- Массовый рефакторинг всей кодовой базы.
- Смена визуального стиля сайта целиком.
- Изменение контактных данных, оффера, юридических текстов.
- Подключение сторонних скриптов/пикселей/виджетов.

---

## 10) Синхронизация
Если над репозиторием могут работать несколько ассистентов или сред
(Codex, Claude Desktop, GitHub web/editor), синхронизация нужна не перед
каждой мелкой задачей, а при передаче работы или риске устаревшей локальной
копии.

Синхронизацию нужно выполнить в случаях:
- новая сессия ассистента;
- переключение между Codex и Claude;
- пауза в работе больше 30–60 минут;
- пользователь явно говорит, что изменения были внесены в GitHub/облаке;
- перед началом крупной задачи или перед финальной проверкой перед публикацией.

Порядок синхронизации:
1. Проверить `git status`.
2. Если рабочее дерево чистое — выполнить `git fetch` и проверить расхождение с remote.
3. Если локальная ветка отстаёт — выполнить `git pull --ff-only`.
4. Если есть локальные незакоммиченные изменения — не делать pull автоматически, сначала разобраться с пользователем.
5. В рамках непрерывной работы одного ассистента над несколькими мелкими задачами повторная синхронизация не обязательна.
