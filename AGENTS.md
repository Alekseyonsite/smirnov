# AGENTS.md — правила редактирования сайта через Claude/Codex

Область действия: весь репозиторий.

## 1) Цель документа
Этот файл задаёт единые правила для AI-ассистентов (Claude/Codex), чтобы правки были:
- предсказуемыми,
- безопасными для SEO и продакшена,
- удобными для ревью человеком.

---

## 2) Общие принципы работы
1. Делай **минимально достаточные** изменения под задачу (без лишних рефакторингов).
2. Сохраняй текущую архитектуру статического сайта (HTML/CSS/JS без фреймворков), если явно не просят иначе.
3. Не меняй тексты/офферы/позиционирование без явного запроса.
4. Не удаляй рабочий функционал ради «красоты кода».
5. Если задача неочевидна — добавь краткие комментарии в PR (что изменено и почему).

---

## 3) SEO/мета-правила (обязательно)
Для каждой новой страницы обязательно:
- уникальный `<title>`;
- уникальный `<meta name="description">`;
- `<link rel="canonical" ...>`;
- Open Graph минимум: `og:type`, `og:url`, `og:title`, `og:description`, `og:image`;
- корректный `twitter:card`.

Для контентных страниц:
- блог: добавлять `BlogPosting` JSON-LD;
- кейсы: добавлять `Article` или `CreativeWork` JSON-LD.

После добавления/удаления страниц:
- обновить `sitemap.xml` (URL + `lastmod`),
- проверить `robots.txt` (актуальная ссылка на sitemap).

---

## 4) Правила по HTML/CSS/JS
1. Предпочитать существующие стили и паттерны из `assets/css/main.css` и текущих шаблонов.
2. Не добавлять инлайн-стили, если можно решить через класс.
3. Не дублировать крупные CSS-блоки между страницами.
4. Не подключать новые библиотеки JS без явного запроса.
5. Любой новый JS должен быть простым, без скрытых сайд-эффектов и без поломки существующей навигации/модалок.

---

## 5) Privacy/analytics/security
1. Не удалять и не ломать Яндекс.Метрику без явного запроса.
2. Если меняются формы/сбор данных — убедиться, что это не противоречит `privacy/index.html`.
3. При работе с `_headers`:
   - не ослаблять существующие заголовки безопасности;
   - кэш-правила должны соответствовать фактическим расширениям файлов.

---

## 6) Git и формат изменений
1. Один PR = одна логическая задача.
2. Commit message: коротко и по делу (`feat: ...`, `fix: ...`, `docs: ...`).
3. В PR-описании обязательно:
   - что сделано;
   - зачем;
   - какие файлы затронуты;
   - как проверяли.

---

## 7) Чеклист перед завершением задачи
Перед финалом AI-ассистент обязан проверить:
- [ ] Нет сломанных внутренних ссылок.
- [ ] На странице ровно один `h1`.
- [ ] `title`/`description`/`canonical` заполнены корректно.
- [ ] Для новых материалов обновлён `sitemap.xml`.
- [ ] Не добавлены случайные инлайн-стили и дубли CSS.
- [ ] Изменения минимальны и соответствуют задаче.

---

## 8) Что запрещено делать без явного запроса
- Массовый рефакторинг всей кодовой базы.
- Смена визуального стиля сайта целиком.
- Изменение контактных данных, оффера, юридических текстов.
- Подключение сторонних скриптов/пикселей/виджетов.
